2022年12月，我作为系统架构设计师主导了某证券交易系统的架构设计与实施。该项目旨在构建一个高性能、高可用的证券业务系统。涵盖用户管理、实时行情交易、智能风控、清结算等核心模块。针对证券行业对高并发、低延迟及数据强一致性的严苛要求，我采用云原生微服务架构，结合分布式事务和Kubernetes弹性扩缩容机制，确保系统稳定运行。该项目总投入550万人民币，历时11个月，于2023年10月正式交付运行至今，显著提升了业务处理效率和系统可靠性。本文根据面向对象分析的两大模型：用例模型和分析模型来描述面向对象分析在该系统中的具体应用。

随着我国资本市场的快速发展，证券行业在2022年迎来业务量的爆发式增长。然而，公司原有的证券交易系统采用传统的单体架构，在业务高峰期暴露出严重的性能问题，特别是在季度结息日等特殊时段，系统并发请求数量激增，导致数据库连接池耗尽，交易延迟大大增加，此外，原有系统还存在扩展性差、模块耦合度高、发布风险大的问题，严重制约了业务发展。

基于上述问题，公司决定投入550万人民币对原有系统进行全面重构。该项目于2022年12月正式启动，历时11个月，于2023年10月成功上线。作为该项目的系统架构设计师，我主要负责整体架构设计、关键技术选型和性能优化等工作。在架构设计方面，我采用云原生架构，通过领域驱动设计方法对系统进行微服务拆分，将原有单体应用划分为多个微服务，包括但不限于用户服务、交易服务、风控服务和清结算服务等独立模块。在技术选型上，通过云原生微服务架构，结合Kafka实现行情推送；采用Redis集群缓存实时行情数据；通过MySQL分库分表和ClickHouse列式存储满足不同业务场景的数据存储需求。在技术架构上，前端采用React+TypeScript+WebSocket实现高性能行情展示，后端基于Spring Boot构建微服务，通过Kubernetes容器编排实现弹性扩缩容，并借助Jenkins实现持续部署，最终打造出一个高性能、高可用的新一代证券交易系统。

在面向对象的用例建模过程中，我们依次完成了识别参与者、合并需求获得用例与细化用例三个步骤。

首先识别系统中的主要参与者，包括普通用户、风控管理员、清结算人员和外部行情服务。普通用户负责开户、下单、查询等日常交易操作；风控管理员负责订单审批和风控策略配置；清结算人员处理清结算流程和差错修复；外部行情服务提供实时和历史行情数据。

在参与者识别的基础上，我们进一步将业务需求整合为高层用例。账户相关的操作，如开户、销户、出入金和持仓转入转出，被统一归入用户管理用例；交易相关的操作，包括下单、查询订单、取消订单和成交信息查询，归入交易管理用例；与审批相关的功能，如风控审核、限额设置和审批流程配置，被整合为风控管理用例；行情数据相关的功能，包括实时行情、历史行情和行情推送，合并为行情管理用例；结算与异常处理类需求，如清算对账、各类结算、交易差错和对账退款等，则被统一纳入清结算管理用例。

在细化用例阶段，我们主要描述了不同用例之间的结构性关系，以增强对业务流程的建模与表达能力。以“提交订单”为例，该用例与“风控检查”和“生成订单记录”之间存在包含关系，表示在提交订单的过程中，系统必须完成风控校验并生成订单数据。若订单提交失败，则会触发“通知用户失败原因”这一用例，两者之间存在扩展关系，仅在失败场景下才执行扩展行为。同时，“提交限价单”和“提交市价单”是“提交交易订单”的具体实现形式，三者之间存在泛化关系。

基于上述用例模型，我们进一步转向分析模型的构建阶段，聚焦于系统内部结构与行为的抽象。为此，我们首先定义了概念类：
1. 实体类：代表系统中的核心业务对象。主要的实体类有：用户（User）、订单（Order）、结算记录（SettlementRecord）、成交信息（ExecutionInformation）、交易工作流（TradeWorkFlow）和事件（WorkflowEvent）。
2. 控制类：负责实现业务逻辑的类。主要的控制类有：OrderController（处理用户下单请求）、SettlementController（处理清结算流程）、QueryController（处理查询请求）。
3. 边界类：用于接收输入并与控制类交互的类。主要的边界类有：OrderAPI（接收下单请求）、WebSocketGateway（推送行情数据）。
接下来是类之间的关系：
1. 依赖关系：描述两个事物中一个事物发生变化会影响另一个事物，例如，OrderController 依赖于 Order，因为它需要操作订单数据。
2. 泛化关系：描述一种特殊一般的关系，例如，订单（Order） 和限价单/市价单之间的泛化关系。
3. 聚合关系：描述了整体和部分的关系，但部分可以独立存在，例如，用户（User） 可以拥有多个订单（Order），但订单独立于用户生命周期存在。
4. 组合关系：描述了整体和部分的关系，但部分不能独立于整体存在，例如，TradeWorkFlow 由多个 WorkflowEvent 组成，事件不能独立存在。
为类添加职责时，以 Order 类为例，属性包括：orderId、userId、status、symbol、orderType、orderSide、quantity等；方法包括：validate()、submit()、cancel()。
为更好刻画订单对象的生命周期和行为变化，我们采用状态图对其状态转移过程进行建模。订单的生命周期包括多个状态和事件触发：
1. 待风控审核：风控审核机制会对订单进行检查。如果风控审核失败，订单将转入“审核失败”状态；
2. 待提交：若风控审核通过，订单进入“待提交”状态，订单会提交至交易所进行处理；
3. 待成交：提交成功后，订单进入“待成交”状态，系统等待成交信息；
4. 部分成交/全部成交：交易所推送成交信息后，订单状态会更新为“部分成交”或“全部成交”。
